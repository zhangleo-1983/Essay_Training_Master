# PRD

## 1. 文档信息

- 产品名称：青少年写作教练（Teen Writing Coach）
- 当前阶段：MVP 向可试用版本过渡
- 文档目标：明确产品定位、用户价值、需求边界、验收标准与迭代方向

## 2. 背景与问题

9-16 岁学生在写作文时通常面临三个核心问题：

- 不会审题，不知道题目真正要表达什么。
- 有想法但不会组织，缺少结构化写作方法。
- 依赖 AI 代写，短期完成作业，长期写作能力没有沉淀。

市场上常见 AI 写作产品偏向“生成结果”，而不是“训练能力”。本项目的核心差异是将 AI 定位为写作教练，而不是代笔工具。

## 3. 产品愿景

打造一款面向青少年的 AI 写作训练产品，通过阶段化引导、启发式提问和结构化反馈，帮助学生逐步建立独立写作能力。

## 4. 目标用户

### 4.1 核心用户

- 9-16 岁学生
- 需要课内作文训练、应试作文训练或表达能力提升的学生

### 4.2 次级用户

- 家长：关注孩子写作过程与进步情况
- 教师：希望布置题目、查看学生写作过程与结果

## 5. 核心价值主张

- 不代写，强调写作能力训练。
- 将写作过程拆成学生可执行的五个阶段。
- 通过会话、草稿和阶段推进形成完整训练闭环。
- 支持多模型后端接入，便于后续成本、性能和供应商策略切换。

## 6. 产品目标

### 6.1 业务目标

- 验证“AI 写作教练”对学生持续使用是否有吸引力。
- 支撑小规模真实用户试用。
- 为后续家长端、教师端和商业化付费能力打基础。

### 6.2 用户目标

- 能顺利完成一篇作文的五步写作流程。
- 能保存和回看自己的草稿与写作过程。
- 能在修改阶段获得结构化反馈，而不是直接拿到成文答案。

## 7. 当前版本范围

### 7.1 已实现范围

- 单页 Web 写作界面
- 五步写作流程
- 前端对接后端 API
- 后端会话创建、读取、列表、软删除、恢复
- 草稿保存与当前阶段更新
- 服务端多模型网关
- API key 鉴权与用户隔离
- SQLite 持久化
- 消息限流
- 模型调用日志与成本估算

### 7.2 当前未实现范围

- 用户注册登录
- 教师端、家长端
- 题库系统
- 作文评分 rubric 体系
- 运营后台
- 生产环境部署与监控体系

## 8. 功能需求

### 8.1 写作训练

- 用户输入作文题目后创建写作会话。
- 系统默认从“审题立意”阶段开始。
- AI 每轮只提出一个启发式问题。
- AI 不能直接代写整段或整篇作文。
- 用户可以在任意时刻编辑草稿。
- 用户可以切换当前写作阶段。

### 8.2 会话管理

- 支持查看会话列表。
- 支持搜索会话。
- 支持软删除与恢复会话。
- 支持按用户隔离会话访问权限。

### 8.3 模型服务

- 支持 `gemini`、`openai`、`deepseek`、`openai-compatible`。
- 支持统一模型调用入口。
- 支持失败重试。
- 支持服务端保存密钥，不在前端暴露。

### 8.4 运营与治理

- 支持消息限流。
- 支持调用日志留存。
- 支持按 provider / model / user / session 统计用量。
- 支持估算 token 与调用成本。

## 9. 非功能需求

- 安全性：模型密钥必须仅存在于服务端。
- 可维护性：后端应按认证、网关、存储、业务逻辑拆分。
- 可扩展性：支持新增模型 provider。
- 可观测性：必须具备基本调用日志和错误定位能力。
- 数据可靠性：会话与草稿需持久化存储。

## 10. 成功指标

### 10.1 产品指标

- 会话创建成功率
- 五步流程完成率
- 单次训练平均消息轮数
- 草稿平均字数
- 会话次日回访率

### 10.2 技术指标

- `/api/health` 可用率
- 模型调用成功率
- 平均响应时延
- 单用户消息限流命中率
- 单次会话平均成本

## 11. 风险与约束

- 当前鉴权仍为静态 API key，不适合直接作为正式生产登录方案。
- 当前数据库为 SQLite，适合单机和小规模试用，不适合高并发多实例写入。
- 目前尚无内容安全、审计和家长/教师权限体系。

## 12. 验收标准

- 仓库内存在完整的后端接口说明、架构说明、数据库说明和部署说明。
- 后端支持至少一种非 Gemini provider 的完整配置与调用路径。
- 会话、日志和限流策略均有文档说明。
- README 能作为项目入口页，指向正式文档。

## 13. 后续版本方向

- V0.2：补齐学生单人版 MVP 的联调验证和体验收尾。
- V0.3：增加题库、写作 rubric 和历史记录。
- V0.4：增加用户体系、教师端与家长端。
- V1.0：形成可运营、可计费、可观测的正式产品。
